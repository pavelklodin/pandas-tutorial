# Описание архитектуры задачи B1

## Хранение исходных данных 

### Immutble исходные данные 

Хранятся в папке 
```
data/input/<year>/<month>/<day> 
```
Пример имени папки 
```
data/input/2025/10/25 
```
Она содержит файлы с информацией о продажах, поступившие в систему _25 октября 2025 года_, обработанные и попавшие в итоговую аналитику (в файл `data/result/revenue_summary.csv`). 

### Метаданные клиентов 

Хранятся в файле 
```
data/input/customers.csv 
```
### Загруженные новые данные для обработки 

Хранятся в папке 
```
/data/input/to_process 
```
Формат имени файла 
```
orders-<year>-<month>-<day>-000x.csv 
```
Где `x` - порядковый номер файла (если таких файлов за день поступило несколько, начинается с нуля). Будем считать, что больше _1000_ файлов за день нам не поступит - это гарантия, звучит разумно. 

Пример имени файла 
```
orders-2025-10-25-0000.csv 
```
Это первый по очереди файл с информацией о продажах, поступивший в систему _25 октября 2025 года_. 

Дата указывает на поступление данных в систему. Данные могут содержать как информацию о продажах за прошлые даты (_late arrivals_), так и информацию, уточняющую ранее внесенные в систему продажи. 

### Обновления метаданных клиентов 

Хранятся в файле 
```
data/input/to_process/customers.csv 
```
## Хранение промежуточных результатов 

Хранятся в файле 
```
data/current_intermediate/enriched_raw_data.csv 
```
Это файл с очищенными и сджойненными данными (заказы и клиенты), обогащенный метриками: 
- `revenue`;
- `order_month`. 

## Хранение итоговой аналитики 

Хранится в файле 
```
data/result/revenue_summary.csv 
```
Это файл `enriched_raw_data.csv`, сгруппированый по:
- `order_month`;
- `segment` внутри `order_month`.

и обогащенный метриками:
- `cumulative_revenue`;
- `rolling_avg_revenue`. 

для каждого сегмента и строками с total'ами по каждому месяцу. Строки вставляются после группы сегментов месяца. 

## Устойчивость алгоритма к сбоям 

### State management 

Осуществляется:
- Раздельным хранением уже обработанных и еще не обработанных файлов; 
- Перемещением необработанных файлов к обработанным после завершения _analytic pipeline_. 

### Реакция на сбои и прерывание _analytic pipeline_ 

- В этом случае недообработанные файлы остаются в исходной папке. При следующем запуске скрипт их обработает. 
- Если какие-либо из данных уже ранее обрабатывалсь, то это не приведет к сбоям в работе, поскольку они буду считаны, как дублированные и отброшены при очистке исходных данных, до совершения всех вычислений. 

## Обработка данных

### Режим запуска скрипта

Считаем, что **скрипт запускается раз в сутки**, и для простоты больше файлов с новыми данными в эти сутки не будет. Это очень легко организовать, **запуская скрипт по cron'у в 23:50**, например.

### Методика расчета нетривиальных метрик (строки ALL в итоговой аналитике)

#### Колонки и их логика
- **Total Revenue (месяца)**: Сумма выручки всех сегментов за текущий месяц.
- **Cumulative Revenue (ALL)**: Накопительный итог всей выручки (total_revenue) с самого начала (января) до текущего месяца.
- **Rolling Avg Revenue (ALL)**: Среднее значение total_revenue за последние 3 месяца (текущий + 2 предыдущих).

#### Январь 2024 (тестовые данные)
- **Выручка по сегментам**: Enterprise (500) + SMB (200) = 700.
- **Cumulative**: Только первый месяц, поэтому просто 700.
- **Rolling Avg**: Так как данных раньше нет, это $\frac{700}{1} = 700$.

#### Февраль 2024 (тестовые данные)
- **Выручка по сегментам**: Enterprise (600) + UNKNOWN (1000) = 1600.
- **Cumulative**: Январь (700) + Февраль (1600) = 2300.
- **Rolling Avg**: Среднее за январь и февраль: $\frac{700 + 1600}{2} = 1150$.

#### Март 2024 (тестовые данные)
- **Выручка по сегментам**: Только SMB (300).
- **Cumulative**: Предыдущий итог (2300) + Март (300) = 2600.
- **Rolling Avg**: Среднее за январь, февраль и март: $\frac{700 + 1600 + 300}{3} = \frac{2600}{3} \approx 866.67$.

### Алгоритм работы _analytic pipeline_
1. **Проверяем pre-conditions**:

   0. Общая **цель** проверки - обеспечить **нормальный запуск** скрипта. В случае невозможности нормально запустить скрипт выдается соответствуюшее сообщение об ошибке. В случае потенциально опасной ситуации выдается предупреждение о необходимости более строгой проверки целостностности данных.
   
      Проверяется общая целостность данных (наличие или отсутствие неоходимых файлов).

      Соответствие исходных данных результату **не проверется**!
   1. Должны существовать **метаданные кастомеров**.
      - обработанные данные `data/input/customers.csv`

        **ИЛИ**
      - данные, которые еще нужно обработать `data/input/to_process/customers.csv`

      Если их **нет** - работа завершается с **ошибкой** (нет метаданных кастомеров).
   2. Должны существовать **данные по заказам для обработки**.

      Если **нет** `/data/input/to_process/orders-<year>-<month>-<day>-000x.csv` - работа **завершается** с сообщением, что нечего обрабатывать (**не ошибка!**).
   3. В заключении проверяется **целостность обработанных данных**.
      - Если **обработанные данные отсутствуют** (папки `data/input/<year>/<month>/<day>` - пусты) - значит это первый старт и промежуточного результата (файл `data/current_intermediate/enriched_raw_data.csv`) **не может существовать**. Поэтому:
        - Если файла `data/current_intermediate/enriched_raw_data.csv` **нет**, проверка завершена **успешно** (первый старт).
        - Если файл `data/current_intermediate/enriched_raw_data.csv` **есть**, работа продолжается и **логируется предупреждение**, что вероятно протеряна часть данных. Рекомендуется ревью и восстановление утеряных данных из backup'а.
      - Если **обработанные данные присутствуют** (папки `data/input/<year>/<month>/<day>` - не пустые) - значит это обычный (не первый) старт и промежуточный результат (файл `data/current_intermediate/enriched_raw_data.csv`) **обязан существовать**. Поэтому:
        - Если файл `data/current_intermediate/enriched_raw_data.csv` **есть**, проверка завершена **успешно** (обычный старт).
        - Если файла `data/current_intermediate/enriched_raw_data.csv` **нет**, работа **завершается** с ошибкой, что потерян промежуточный результат. Рекомендуется переместить обработанные данные в папку `/data/input/to_process/` и перезапустить скрипт для восстановления промежуточного результата.
2. **Считываем метаданные клиентов** из файла
   ```
   data/input/customers.csv 
   ```
   в `DataFrame` 
   ```
   current_customers_df 
   ```
   `DataFrame` с метаданными клиентов нам будет нужен в любом случае, для приведения формата новых данных к промежуточным результатам. 
3. **Если есть обновленные метаданные клиентов** в файле
   ```
   data/input/to_process/customers.csv
   ```
   - Считываем обновленные метаданные клиентов из файла 
     ```
     data/input/to_process/customers.csv 
     ```
     в `DataFrame` 
     ```
     customers_to_process_df
     ```
   - Конкатенируем `current_customers_df` и `customers_to_process_df` в `customers_df`
   - Если данные дублируются - оставляем последнюю запись, даже если раньше у клиента был сегмент, а теперь - не стало.
   - Сохраняем обновленные метаданные клиентов в
     ```
     data/input/customers.csv.tmp 
     ```
   - Заменяем временный файл на постоянный
     ```
     data/input/customers.csv
     ```
     с помощью
     ```
     os.replace(tmp, target) 
     ```
   - Удаляем
     ```
     data/input/to_process/customers.csv 
     ```
4. **Считываем промежуточные результаты** из файла 
   ```
   data/current_intermediate/enriched_raw_data.csv 
   ```
   в `DataFrame` 
   ```
   current_orders_df
   ```
5. **Обновляем данные о клиентах**, если нужно
   ```
   current_orders_df.update(customers_df)
   ```
6. **Считываем загруженные новые данные** - все файлы из папки 
   ```
   /data/input/to_process 
   ```
   в `DataFrame`
   ```
   orders_to_process_df
   ```
   Если файлов с новыми данными несколько - читаем в цикле и конкатенируем. 
7. **Обрабатываем и обогащаем** `orders_to_process_df` 
   - Джойним с `customers_df` (или если его нет - с `current_customers_df`) 
   - Очищаем данные 
   - Обогащаем их метриками
     - `revenue`
     - `order_month` 
8. **Конкатенируем** `current_orders_df` и `orders_to_process_df` в `concatenated_orders_df` 
   - **Очищаем** `concatenated_orders_df` в `cleaned_orders_df` и в случае одинаковых `order_id` - оставляем только последнюю запись.
   - **Обновляем** файл с промежуточными результатами
     - Сохраняем `cleaned_orders_df` в файл
       ```
       data/current_intermediate/enriched_raw_data.csv.tmp
       ```
     - Заменяем временный файл на постоянный
       ```
       data/current_intermediate/enriched_raw_data.csv
       ```
       с помощью
       ```
       os.replace(tmp, target)
       ```
9. **Перемещаем файлы с обработанными новыми данными** из папки
   ```
   /data/input/to_process
   ```
   в
   ```
   data/input/<year>/<month>/<day>
   ```
10. **Пересчитываем** `cleaned_orders_df` и `DataFrame` для итоговой аналитики - `final_result_df` 
11. **Сохраняем** `final_result_df` в файл
    ```
    data/result/revenue_summary.csv.tmp
    ```
    - Заменяем временный файл на постоянный 
      ```
      data/result/revenue_summary.csv
      ```
      с помощью
      ```
      os.replace(tmp, target)
      ```
